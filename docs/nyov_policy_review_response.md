# NYOV Policy and Rubric Review

## 1. Risk Review and Policy Gaps
- **Entity vs. Field Confusion**: The current rubric conflates entity identity verification with field-level verification in `verified_strong`. Two sources agreeing on title/artist proves the recording *exists*, but does not guarantee the release date or track number are correct.
- **YouTube Music Vulnerability**: Counting raw YouTube Music video IDs as Tier 1 is dangerous. YouTube contains user-generated content, meaning fan uploads, lyric videos, or live bootlegs could incorrectly validate a studio recording.
- **Fuzzy Match Ambiguity**: The `needs_review` bucket specifies "sources disagree" or "a source match is fuzzy," but does not define the threshold (e.g., string distance or Levenshtein score) that triggers a conflict vs. an accepted minor variation (like capitalization).
- **Subjective Tag Promotion**: The policy lacks explicit guidance on whether `derived_consensus` for subjective fields permits automated promotion, or if subjective fields always require a human click.

## 2. Revised Rubric
The `docs/nyov_verification_policy_rubric.md` file has been updated to address these gaps. Key changes include:
- Explicit separation of Entity Verification from Field Verification.
- Stricter rules for YouTube Music.
- Tier 1 promotion for SecondHandSongs on relational facts.

## 3. Recommended Minimum Schema

For future `verification_attempt` and `promotion` records, the following schema is recommended:

```sql
CREATE TABLE nyov_verification_attempts (
    attempt_id TEXT PRIMARY KEY,
    nyov_id TEXT NOT NULL,
    status TEXT NOT NULL, -- 'verified_strong', 'verified_supported', 'needs_review', 'not_verified'
    evidence_payload_json TEXT NOT NULL, -- Snapshot of the IDs and fields compared
    matched_sources_json TEXT NOT NULL, -- List of sources that agreed
    conflict_details_json TEXT, -- If status is 'needs_review', details on the mismatch
    attempted_at TEXT NOT NULL,
    automation_version TEXT NOT NULL
);

CREATE TABLE nyov_promotions (
    promotion_id TEXT PRIMARY KEY,
    nyov_id TEXT NOT NULL,
    target_table TEXT NOT NULL, -- 'songs', 'recordings', 'external_links', etc.
    target_key TEXT NOT NULL, -- the ID in the target table (e.g., Recording ID)
    promoted_fields_json TEXT NOT NULL, -- Which specific fields were officially written
    promoted_at TEXT NOT NULL,
    promoted_by TEXT NOT NULL, -- e.g., 'codex_auto', 'human_username'
    notes TEXT
);
```

## 4. Recommendation on YouTube Music
**Recommendation**: YouTube Music should **only** count as Tier 1 under strict conditions:
- The ID is a stable YouTube Music Song/Album entity ID.
- If it is a Video ID, the video must be an "Art Track" (auto-generated by YouTube, usually bearing "Provided to YouTube by [Label]").
- Official Artist Channel (OAC) official music videos can be Tier 2 for metadata, Tier 1 for linking.
- Fan uploads, lyric videos, and live bootlegs must remain Tier 3.

## 5. Recommendation on SecondHandSongs
**Recommendation**: SecondHandSongs should be elevated to **Tier 1 for cover and original-song relationship facts**.
SecondHandSongs is a highly moderated, specialized database dedicated to tracking musical works and performances. Its performance lineage data is often more accurate and comprehensive than generalist databases like Spotify or even MusicBrainz. However, for standard recording metadata (ISRC, release date), it should remain Tier 2.
