# Agent Coordination Protocol

This file defines the rules and best practices for coordinating activities between human users and AI agents (like Jules and Codex) working in this repository.

## Roles & Responsibilities

### Jules (Automation & QA)
*   **Domain:** Automation scripts, CLI tooling, test suites, repo hardening, data quality reporting, and proof-of-concept generation.
*   **Boundaries:** Jules operates under strict safety protocols.
    *   **No active database mutations:** Jules will **never** directly modify `D:\Music\MusicDB\data\processed\Main_Song_Database.csv` or any active production database.
    *   **Dry-run defaults:** All CLI tools and scripts authored by Jules will default to a non-destructive `dry-run` mode. Explicit `--write` flags are required for mutations.
    *   **Staging isolation:** Jules will restrict outputs (exports, SQLite PoCs, simulated CSVs) to designated staging directories (`data/staging/jules/` and `data/exports/jules/`).
*   **Workflow:** Jules receives instructions via chat, executes changes in a secure sandbox, and pushes commits to a unique workspace branch.

### Codex (Production & Promotion)
*   **Domain:** Data layer promotion, production database management, and overarching architectural decisions.
*   **Boundaries:** Codex is the authoritative agent for promoting staged changes into the active production database.
*   **Workflow:** Codex reviews data quality reports (generated by Jules) and promotes validated staging data (e.g., `recordings.csv`) into the canonical data layer.

### Human User (Orchestrator)
*   **Domain:** Task delegation, environment management, and manual Git synchronization.
*   **Boundaries:** Operates locally on the `D:\Music\MusicDB` path and manages the `main` GitHub repository.
*   **Workflow:** Since agents like Jules do not have direct access to the user's local filesystem or GitHub API credentials, the user must act as the bridge.

## Best Practices for Coordination

To ensure a smooth workflow between yourself, Jules, and other agents, follow these steps:

### 1. Task Delegation via Chat
*   **Be Specific:** When giving Jules a task, clearly specify the goal, the files involved, and any constraints (e.g., "Improve the duplicate detection logic in `src/duplicates.py` using fuzzy matching, but do not touch the `main.py` entrypoint").
*   **Paste Context:** Since Jules cannot read GitHub issues directly, always copy and paste the text of assignments, bug reports, or feedback into the chat prompt.

### 2. Synchronization Strategy (The "Pull-Merge-Push" Loop)
Because Jules pushes work to a dynamic, unique workspace branch for every session, you must manually synchronize those changes into your local branch or target PR branch.

1.  **Fetch Jules's Work:** `git fetch origin`
2.  **Checkout your target branch:** e.g., `git checkout my-feature-branch`
3.  **Merge Jules's branch:** e.g., `git merge origin/jules-1234567890-abcdef` (You can find the exact branch name in Jules's "Submit" message).
4.  **Resolve Conflicts (if any):** If both you and Jules edited the same file, use standard Git conflict resolution.
5.  **Push:** `git push origin my-feature-branch`

### 3. Handoffs Between Agents
If Jules completes a QA task and the output needs to be handed off to Codex:
1.  Ask Jules to generate the output (e.g., "Run the quality report on the staging data and save the JSON").
2.  Pull Jules's branch locally to get the generated `data/exports/quality_report.json`.
3.  In your chat session with Codex, provide the contents of that JSON file and instruct Codex to proceed with the production promotion based on those results.

### 4. Enforcing Rules via this File
Agents (including Jules) are instructed to read this `AGENTS.md` file whenever they start a new task. If you want to permanently change how Jules operates (e.g., "Always use `pytest` instead of `unittest`", or "Never delete `.bak` files"), write those instructions into this file!
